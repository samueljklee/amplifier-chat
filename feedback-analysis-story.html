The Problem

Bug reports are either too much or too little

•••

Too Vague

"It crashed." No logs. No error messages. No reproduction steps. Developers spend more time asking for details than fixing the bug.

>_

Too Much

Full terminal dumps. Entire log files. Screenshots of every window. The signal is buried in 200 lines of noise that developers have to sift through.

?!

Privacy Anxiety

"What if my API keys are in the logs?" Users don't know what's in their session data. So they share nothing — or share everything and worry.

Step 2

The recipe reads three sources

1

Session transcript

Reads transcript.jsonl — looks only for infrastructure failures: tool:error, provider:error, unhandled exceptions. Ignores conversation content entirely.

2

Server logs

Reads serve.log for HTTP 5xx responses, SSE subscriber drops, connection errors, and tracebacks correlated with the session ID.

3

GitHub issue search

For each error pattern found, runs gh issue list to find existing related issues. Rates each match as high, medium, or low confidence.

What it ignores

User messages. Assistant responses. What you asked about. The content of successful tool calls. Anything that isn't a platform-level failure. The prompt is explicitly scoped to infrastructure only.

Under the Hood

A declarative recipe defines the pipeline

name: "feedback-analysis" steps: - id: "github-search" agent: foundation:explorer prompt: | Extract error signatures from transcript... Search GitHub for each pattern... Rate confidence: high / medium / low output: github_findings - id: "log-analysis" agent: foundation:explorer prompt: | Find ALL errors in transcript + server logs... Merge with {{github_findings}} from step 1... Return unified JSON array output: all_findings

Two steps. Step 1 searches GitHub for known issues. Step 2 reads all logs and merges everything into a single findings array. The output feeds directly into the UI.

Step 3

Every finding gets a checkbox

Findings are grouped into three categories: Related Issues, Session Errors, and Server Logs. Each one is shown with enough context to understand — but no more.

Every finding starts checked. The user unchecks anything they don't want in the report. Expandable details let them inspect stack traces and log context before deciding.

Low-confidence GitHub matches are automatically filtered out. Only high and medium confidence issues make the cut.

Privacy by Design

Three layers of protection

Scoped prompt

The analysis prompt is explicitly instructed to look ONLY for infrastructure failures. It ignores user messages, assistant responses, and conversation content.

Checkbox consent

Every finding is presented individually with expandable details. Users review each one and uncheck anything they don't want shared. Nothing is auto-sent.

Local execution

The entire analysis runs locally on the user's machine. The hidden analysis session uses the same local Amplifier daemon. No data leaves until the user clicks Submit.

Architecture

How the pieces connect

feedback-widget.js # Frontend — modal UI, SSE listener ↓ POST /chat/api/feedback/analyze feedback.py # Backend — creates hidden session ↓ Creates analysis session (hidden: true) ↓ Builds scoped analysis prompt ↓ Kicks off background execution amplifierd daemon # Runs analysis via LLM + tools ↓ Reads transcript.jsonl ↓ Reads serve.log ↓ Runs gh issue list ↓ Returns JSON array of findings feedback-widget.js # Receives via SSE stream ↓ Parses findings, filters low-confidence ↓ Renders checkboxes per finding ↓ User reviews, unchecks, submits GitHub # Pre-filled issue with checked items only

The analysis session is marked hidden: true so it never appears in the user's session history. It runs in the background while they type their description.

Design Decisions

Why it works this way

?

Why a hidden session instead of a direct API call?

The analysis needs to call tools (read_file, bash, grep) and reason about errors. That's exactly what an Amplifier session does. A hidden session reuses the full agent loop — no special infrastructure needed.

?

Why filter low-confidence GitHub matches?

Keyword overlap creates false positives. "Timeout" matches dozens of issues. Only high and medium matches are actionable — low confidence just creates noise.

?

Why checkboxes instead of a summary?

Summaries hide detail. Checkboxes give the user granular control. They can exclude an unrelated bash timeout while keeping the provider error. The user is the final arbiter.

?

Why open GitHub instead of sending directly?

Opening GitHub lets the user review the issue one more time before creating it. They can edit the title, add context, or cancel. No surprises, no auto-filing.
